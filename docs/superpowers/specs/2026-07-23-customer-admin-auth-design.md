# 고객·관리자 인증 설계

**작성일**: 2026-07-23  
**상태**: 설계 승인됨

## 결정

- 고객은 이메일 기반 회원가입, 로그인, 로그아웃, 비밀번호 재설정 흐름을 사용한다.
- 관리자는 회원가입할 수 없고 관리자 로그인만 사용한다.
- 초기 `super_admin` 계정은 현재 코드, seed, 환경변수 또는 브라우저 저장소에 만들지 않는다.
- 향후 Supabase 연동 시 초기 슈퍼 계정은 배포 운영 절차로 생성한다.
- `super_admin`만 기존 고객 계정에 `admin` 권한을 부여하거나 회수할 수 있다.
- 현재 단계의 목업 인증은 화면과 상태 전이 검증용이며 보안 경계로 표현하지 않는다.

## 역할과 접근

| 역할 | 가입 | 접근 |
|---|---|---|
| 비회원 | 없음 | 공개 페이지, 고객 로그인·회원가입 |
| customer | 공개 회원가입 | 공개 페이지, 고객 계정 화면 |
| admin | 공개 가입 불가 | 관리자 로그인 후 관리자 화면 |
| super_admin | 운영 단계에서만 생성 | 관리자 화면과 관리자 권한 관리 |

클라이언트가 전달한 역할값은 신뢰하지 않는다. 실제 운영에서는 모든 관리자 경로와 권한 변경을 서버 세션과 저장소 권한 정책으로 다시 확인한다.

## 화면 구조

- `/login`: 고객 로그인
- `/signup`: 고객 회원가입
- `/forgot-password`: 고객 비밀번호 재설정 요청
- `/account`: 고객 계정 기본 화면
- `/admin/login`: 관리자 로그인
- `/admin/users`: 슈퍼 관리자 전용 권한 관리 화면. 현재 목업에서는 잠금 안내 상태만 제공한다.

고객 인증 화면은 고객 사이트의 헤더·카드·입력·버튼 스타일을 사용한다. 관리자 로그인은 관리자 화면의 높은 정보 밀도와 blue/neutral 토큰을 사용하되 동일한 입력·버튼 컴포넌트를 재사용한다.

## 인증 경계

UI는 `AuthRepository` 계약만 사용한다. 현재 목업 어댑터는 세션 상태 전이를 재현하고, 이후 Supabase 어댑터가 회원가입, 로그인, 로그아웃, 비밀번호 재설정과 세션 조회를 대체한다.

운영 전환 시 필요한 서버 책임:

- 암호를 애플리케이션 코드나 브라우저 저장소에 저장하지 않는다.
- 관리자 라우트는 서버에서 세션과 활성 역할을 검사한다.
- 역할 변경은 `super_admin` 전용 서버 동작으로만 허용한다.
- 역할 변경, 로그인 실패, 권한 거부를 감사 기록에 남긴다.
- 회원 존재 여부를 추측할 수 없는 오류 문구와 요청 제한을 적용한다.

## 현재 목업 제한

- 실제 슈퍼 계정과 관리자 계정을 seed하지 않는다.
- 관리자 로그인은 계정 미설정 안내를 제공하며 성공 상태를 위조하지 않는다.
- 고객 목업 계정은 UI 검증용 세션이며 실제 인증 또는 개인정보 저장 기능으로 표시하지 않는다.
- `/admin/users`는 Supabase 연동 전 잠금 상태와 필요한 운영 절차만 보여준다.

## 검증

- 고객 회원가입·로그인·로그아웃·재설정 흐름
- 관리자 회원가입 경로와 링크가 존재하지 않음
- 비관리자와 미인증 사용자의 관리자 접근 차단 상태
- 슈퍼 계정·비밀번호·서비스 키가 소스와 클라이언트 저장소에 없음
- 역할 변경 UI가 슈퍼 관리자 전용으로 잠겨 있음
- 390px·1440px, 키보드, 오류 연결, 비밀번호 표시 전환과 상태 알림
