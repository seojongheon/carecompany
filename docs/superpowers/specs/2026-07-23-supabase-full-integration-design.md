# Supabase 전체 연동 설계

**작성일:** 2026-07-23  
**상태:** 사용자 승인 후 문서 검토 대기  
**범위:** 인증, 권한, 사례, 홈페이지 콘텐츠, 가격, 미디어 메타데이터, RLS, 선택적 Storage 준비

## 1. 목표

현재 브라우저 목업 저장소를 Supabase 기반 영속 저장소로 전환한다. 고객 인증과 공개 콘텐츠 조회를 실제 백엔드에 연결하고, 관리자 콘텐츠 편집과 슈퍼 관리자 역할 관리를 서버 검증 경계 안에서 제공한다.

초기 슈퍼 관리자 자격 증명은 소스 코드, seed, 브라우저 저장소 또는 공개 환경변수에 포함하지 않는다. Supabase Storage는 비용 또는 플랜 제약이 있는 동안 실제 연결하지 않으며, 계정 업그레이드 후 별도 활성화 절차만으로 연결할 수 있게 준비한다.

## 2. 확정된 설계 원칙

- 데이터 구조는 하이브리드 정규화를 사용한다.
- 검색, 관계, 권한과 제약이 중요한 데이터는 관계형 테이블에 저장한다.
- 홈페이지 문구 묶음과 복원용 버전은 `jsonb` 스냅샷으로 저장한다.
- 공개 데이터와 관리자 데이터의 경계는 RLS와 서버 권한 검사로 이중 보호한다.
- 공개 회원가입은 항상 `customer`만 생성한다.
- `admin` 또는 `super_admin`은 공개 회원가입으로 생성하거나 획득할 수 없다.
- 역할 변경은 `super_admin` 전용 보안 정의 함수로만 수행한다.
- Storage 활성화 여부와 관계없이 사례와 콘텐츠 데이터베이스 연동은 완료할 수 있어야 한다.

## 3. 데이터 모델

### 3.1 인증과 권한

- `profiles`
  - `id`: `auth.users.id` 참조
  - `email`: 표시와 운영 조회용 정규화 이메일
  - `role`: `customer | admin | super_admin`
  - `is_active`: 관리자 접근 활성 상태
  - 생성·수정 시각
- `admin_role_audit_logs`
  - 실행자, 대상 사용자, 이전·이후 역할, 이전·이후 활성 상태, 사유, 실행 시각
  - 수정·삭제 불가

Auth 사용자 생성 트리거는 `customer` 프로필만 만든다. 역할 필드는 일반 프로필 수정 경로에서 제외한다.

### 3.2 포트폴리오

- `services`: 고정 서비스 키, 이름, 설명, 정렬, 활성 상태
- `portfolio_cases`: 사례 본문, 공개 상태, SEO, 개인정보 검토 결과, 게시 시각
- `case_media`: 단계, 순서, 대표 여부, 공개 여부, 파일 메타데이터, Storage 경로, 처리 상태
- `case_videos`: YouTube 정보, 순서, 공개 여부
- `tags`: 서비스별 태그
- `case_tags`: 사례와 태그의 다대다 관계

데이터베이스는 서비스 키·슬러그 중복, 사례 슬러그 중복, 외래 키, 사례당 미디어 69개, 영상 3개, 대표 이미지 1개를 강제한다. 게시 RPC는 개인정보 체크리스트, 공개 준비 미디어와 대표 이미지를 한 트랜잭션에서 검증한다.

### 3.3 홈페이지와 가격

- `site_content`
  - 단일 활성 레코드
  - `draft jsonb`, `published jsonb`, 게시 버전과 수정 시각
- `site_content_versions`
  - 게시·복원 가능한 콘텐츠 스냅샷, 작성자와 생성 시각
- `price_items`
  - 서비스 키, 이름, 가격 표시, 조건 배열, 노출 여부와 정렬

홈페이지 게시는 초안 검증, 버전 생성과 게시본 교체를 하나의 트랜잭션으로 수행한다.

## 4. RLS와 서버 권한

### 4.1 익명 사용자

- 활성 서비스 조회 허용
- `published` 사례와 연결된 공개·처리 완료 미디어 및 공개 영상만 조회 허용
- 게시된 홈페이지 콘텐츠와 노출 가격만 조회 허용
- 비공개·삭제 사례, 초안, 원본 경로와 관리자 데이터 접근 거부

### 4.2 고객

- 익명 공개 범위와 동일한 콘텐츠 조회 허용
- 자신의 프로필 조회와 안전한 표시 필드 수정만 허용
- 역할과 관리자 활성 상태 변경 거부

### 4.3 관리자

- 활성 `admin`과 `super_admin`만 사례·미디어 메타데이터·태그·가격·사이트 콘텐츠 관리 허용
- 역할 변경, 감사 로그 변경과 서비스 역할 키 사용 거부

### 4.4 슈퍼 관리자

- 콘텐츠 관리 권한 포함
- 기존 사용자 역할과 관리자 활성 상태 변경 허용
- 모든 역할 변경을 감사 로그에 기록
- 마지막 활성 슈퍼 관리자의 자기 권한 제거 방지

클라이언트가 전달한 역할 값은 신뢰하지 않는다. 정책과 서버 코드는 `auth.uid()`에 대응하는 `profiles` 행을 조회한다.

## 5. 초기 슈퍼 관리자 운영 절차

1. Supabase Auth 관리 화면 또는 서버 전용 관리 API로 사용자를 생성한다.
2. 서비스 역할 키가 필요한 일회성 운영 명령에 대상 사용자 UUID와 변경 사유를 실행 인자로 전달한다.
3. 명령은 대상 존재 여부와 현재 슈퍼 관리자 수를 검증한 뒤 역할을 변경한다.
4. 변경 결과를 `admin_role_audit_logs`에 기록한다.
5. 이메일, 비밀번호, 사용자 UUID 또는 서비스 역할 키를 저장소에 커밋하지 않는다.

초기 슈퍼 관리자는 이번 구현에서 생성하거나 원격 프로젝트에 적용하지 않는다.

## 6. Storage 지연 활성화 설계

Storage 준비물은 핵심 데이터베이스 마이그레이션과 분리한다.

- `case-originals`: 비공개 원본 버킷
- `case-public`: 검토 완료된 파생 이미지 버킷
- 경로: `<case-id>/<media-id>/<version>.<ext>`
- 원본은 활성 관리자만 접근
- 공개 버킷은 `ready`이면서 공개 처리된 레코드만 조회 가능
- 사용자 파일명은 저장 경로나 공개 URL에 사용하지 않음

기본 환경에서는 `SUPABASE_STORAGE_ENABLED=false`로 동작한다. 이 상태에서는 다음을 보장한다.

- 원격 버킷 생성과 업로드를 시도하지 않는다.
- 관리자 업로드 화면은 "Storage 연결 대기" 상태를 명확히 표시한다.
- 이미지 메타데이터 저장이나 공개 처리를 성공으로 오인하지 않는다.
- 기존 목업 미디어는 개발 환경에서만 표시 가능하다.

버킷 생성 SQL, Storage RLS, 업로드 어댑터, MIME·크기 검증과 운영 체크리스트는 별도 활성화 패키지로 준비한다. 계정 업그레이드 후 관리자가 체크리스트에 따라 버킷 설정을 적용하고 서버 환경변수를 활성화하면 코드 변경 없이 연결된다.

## 7. 애플리케이션 구조

- 브라우저 클라이언트: 고객 인증 상태와 공개 읽기
- 서버 클라이언트: 쿠키 기반 세션 검증과 사용자 범위 작업
- 관리 클라이언트: 운영 스크립트 전용, 애플리케이션 런타임과 분리
- 미들웨어 또는 서버 레이아웃: `/admin` 세션과 역할 검사
- 서버 액션 또는 Route Handler: 관리자 변경, 게시, 역할 변경과 업로드 서명

기존 `AuthRepository`와 UI 경계는 유지하고 Supabase 구현체로 교체한다. 포트폴리오와 사이트 콘텐츠 저장소는 비동기 Supabase 계약으로 전환하며, 공개 페이지는 서버에서 게시 데이터만 읽어 렌더링한다.

## 8. 데이터 전환

- 브라우저 `localStorage` 데이터는 자동 업로드하지 않는다.
- 목업 이미지, 객체 URL, 개인정보 미검토 콘텐츠는 마이그레이션 대상에서 제외한다.
- 필요한 경우 별도의 일회성 검토 도구로 승인된 JSON만 가져온다.
- 네 개의 서비스 키는 안정적인 고정 값으로 seed한다.
- seed에는 사용자, 관리자 또는 슈퍼 관리자 계정을 포함하지 않는다.

## 9. 오류와 보안 처리

- 로그인과 비밀번호 재설정은 계정 존재 여부를 노출하지 않는 공통 오류를 사용한다.
- 인증 누락과 권한 부족은 서버에서 각각 `401`, `403`으로 구분한다.
- 게시와 역할 변경은 실패 시 전체 롤백한다.
- 네트워크 실패 시 사용자 입력을 화면에 유지하고 성공 상태를 표시하지 않는다.
- 서비스 역할 키와 데이터베이스 비밀번호는 서버 전용 환경변수로만 사용한다.
- 로그에 토큰, 비밀번호, 원본 이미지 경로와 고객 개인정보를 남기지 않는다.

## 10. 검증 전략

### SQL 및 RLS

- 익명 사용자의 비공개·삭제 사례 조회와 모든 변경 거부
- 고객의 다른 프로필 및 관리자 데이터 접근 거부
- 고객과 관리자의 역할 필드 변경 거부
- 활성 관리자의 콘텐츠 관리 허용
- 비활성 관리자의 관리자 작업 거부
- 관리자의 역할 관리 거부
- 슈퍼 관리자의 역할 부여·회수와 감사 로그 생성
- 마지막 슈퍼 관리자 보호
- Storage 비활성 상태에서 업로드 시도 차단
- Storage 활성화 후 원본 비공개와 공개 파생본 조회 정책 검증

### 애플리케이션

- 고객 회원가입, 로그인, 로그아웃과 재설정
- 익명·고객의 `/admin` 서버 차단
- 관리자 콘텐츠 편집·게시
- 슈퍼 관리자 권한 관리
- 공개 페이지에서 초안과 비공개 데이터가 노출되지 않음
- Storage 비활성 UI와 활성화 전환 구성 검증

## 11. 산출물

- Supabase CLI 설정과 환경변수 예시
- 핵심 스키마·함수·트리거·RLS 마이그레이션
- 선택적 Storage 설정 SQL과 활성화 체크리스트
- 데이터베이스 seed와 생성 타입
- 브라우저·서버·운영 Supabase 클라이언트
- 인증, 포트폴리오와 사이트 콘텐츠 Supabase 어댑터
- 서버 관리자 가드와 역할 관리 작업
- RLS, 단위, 컴포넌트와 E2E 테스트
- 초기 슈퍼 관리자와 Storage 활성화 운영 문서

## 12. 범위 제외

- 실제 초기 슈퍼 관리자 생성
- 원격 프로젝트에서의 역할 승격 실행
- 플랜 확인 전 Storage 버킷 생성 또는 실제 파일 업로드
- 이미지 변환 작업자와 CDN 최적화 서비스의 실제 배포
- 검토되지 않은 브라우저 목업 데이터의 자동 이전
