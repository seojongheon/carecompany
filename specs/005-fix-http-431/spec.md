# Feature Specification: 로그인 후 HTTP 431 복구

**Feature Branch**: `main`

**Created**: 2026-07-23

**Status**: Approved

**Input**: 로그인 후 홈페이지 요청이 `431 Request Header Fields Too Large`로 실패하는 문제를 재현하고, 수정계획을 문서화한 뒤 실제 접속이 가능하도록 수정한다.

## User Scenarios & Testing

### User Story 1 - 로그인 후 홈페이지 재접속 (Priority: P1)

사용자는 인증 쿠키가 포함된 상태에서도 홈페이지와 관리자 화면을 정상적으로 열 수 있다.

**Why this priority**: 현재는 로그인 후 모든 페이지 진입이 HTTP 계층에서 차단되므로 서비스 사용 자체가 불가능하다.

**Independent Test**: 32KB 요청 헤더를 포함해 홈페이지를 요청했을 때 애플리케이션이 431이 아닌 정상 응답을 반환하는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 인증 과정에서 여러 쿠키 청크가 남은 브라우저, **When** 전체 요청 헤더가 16KB를 초과하지만 64KB 이하인 상태로 홈페이지를 요청하면, **Then** 서버는 요청을 애플리케이션에 전달한다.
2. **Given** 정상적인 브라우저, **When** 홈페이지·로그인·관리자 로그인 페이지를 요청하면, **Then** 기존과 동일하게 정상 응답한다.
3. **Given** 64KB를 초과하는 비정상 요청, **When** 홈페이지를 요청하면, **Then** 서버는 요청을 거부한다.

### Edge Cases

- 개발 서버가 설정 변경 전에 실행 중이면 새 한도가 적용되지 않으므로 재시작이 필요하다.
- 브라우저에 오래된 Supabase 쿠키 청크가 남아도 요청이 64KB 이하이면 프록시까지 도달해 기존 `tokens-only` 정책으로 갱신될 수 있어야 한다.
- 64KB보다 큰 헤더는 메모리 소모 공격 표면을 제한하기 위해 허용하지 않는다.

## Requirements

### Functional Requirements

- **FR-001**: 개발 서버와 자체 호스팅 시작 서버는 최대 65,536바이트의 HTTP 요청 헤더를 처리해야 한다.
- **FR-002**: 32KB 크기의 회귀 요청은 431이 아닌 애플리케이션 응답을 받아야 한다.
- **FR-003**: 65,536바이트를 넘는 요청은 계속 거부되어야 한다.
- **FR-004**: 기존 Supabase `tokens-only` 쿠키 인코딩과 관리자 권한 검증은 변경하지 않아야 한다.
- **FR-005**: 설정 변경 후 개발 서버를 재시작하고 실제 localhost 요청으로 수정 여부를 검증해야 한다.

## Success Criteria

### Measurable Outcomes

- **SC-001**: 17KB 및 32KB 헤더 요청이 localhost 홈페이지에서 HTTP 200을 반환한다.
- **SC-002**: 70KB 헤더 요청은 HTTP 431을 반환한다.
- **SC-003**: 전체 단위·컴포넌트 테스트, 린트, 타입 검사와 프로덕션 빌드가 모두 통과한다.
- **SC-004**: 정상 헤더로 홈페이지·로그인·관리자 로그인 페이지를 요청했을 때 HTTP 200을 반환한다.

## Assumptions

- 로컬 실행과 자체 호스팅 실행은 `pnpm dev` 또는 `pnpm start`를 사용한다.
- 외부 호스팅 플랫폼이 별도 프록시 한도를 적용할 경우 그 한도는 플랫폼 설정에서 별도로 관리한다.
- 고객의 현재 실패 요청은 16KB 초과 64KB 이하 범위에 있다.
