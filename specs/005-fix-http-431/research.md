# Research: HTTP 431 발생 계층과 복구 경계

## Decision 1: 발생 계층

**Decision**: 431은 Next.js 프록시와 라우팅 이전의 Node HTTP 파서에서 발생한다.

**Rationale**: 현재 Node 런타임의 `http.maxHeaderSize`는 16,384바이트이며 동일 서버에 보낸 16KB 요청은 200, 17KB 요청은 431을 반환했다. 따라서 애플리케이션 내부에서 쿠키를 삭제하거나 리다이렉트를 변경해도 해당 요청에는 실행 기회가 없다.

**Alternatives considered**: 로그인 성공 리다이렉트 변경, 프록시에서 쿠키 삭제, 브라우저 스토리지 재삭제. 세 방법 모두 HTTP 파서 이후 단계이므로 최초 431을 복구하지 못한다.

## Decision 2: 제한값

**Decision**: 개발·시작 런타임의 최대 요청 헤더를 65,536바이트로 설정한다.

**Rationale**: Supabase 쿠키 청크와 일반 브라우저 헤더가 일시적으로 16KB를 넘는 상태를 복구하면서도 상한을 유지할 수 있다. 무제한 허용은 하지 않는다.

**Alternatives considered**: 32KB는 실제 브라우저의 정확한 헤더 크기를 알 수 없어 복구 여유가 작다. 128KB 이상은 필요한 범위보다 공격 표면을 크게 넓힌다.

## Decision 3: 쿠키 정책

**Decision**: 기존 `tokens-only` 인코딩을 유지하고 별도 인증 로직 변경은 하지 않는다.

**Rationale**: 새 세션이 다시 과도하게 커지는 원인은 이미 제거되어 있다. 이번 변경은 남아 있거나 부가 헤더와 합쳐져 16KB를 넘는 요청이 애플리케이션에 한 번 도달할 수 있도록 하는 복구 경로다.
