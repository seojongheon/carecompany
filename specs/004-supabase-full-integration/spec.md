# Feature Specification: Supabase 전체 연동

**Feature Branch**: `004-supabase-full-integration`

**Created**: 2026-07-23

**Status**: Approved

**Input**: 연결된 Supabase 프로젝트에 데이터베이스를 설계하고, 인증·RLS·관리자 권한·사례·홈페이지 콘텐츠·가격을 실제 애플리케이션과 연동한다. Storage는 비용 조건이 확인될 때까지 실제 연결하지 않고 즉시 활성화 가능한 준비 상태로 둔다.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 안전한 공개 콘텐츠 조회 (Priority: P1)

방문자는 관리자가 게시한 서비스, 사례, 가격과 홈페이지 콘텐츠만 볼 수 있다. 비공개·삭제·초안 데이터는 직접 주소를 알아도 노출되지 않는다.

**Why this priority**: 공개 사이트의 정확성과 개인정보 보호가 서비스의 핵심 가치이며 모든 관리자 기능의 결과가 이 경계로 전달된다.

**Independent Test**: 익명 상태에서 공개 사례는 조회되고 비공개 사례·초안·관리자 데이터는 조회되지 않는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 게시된 사례와 비공개 사례가 존재할 때, **When** 익명 방문자가 목록과 상세 주소를 조회하면, **Then** 게시된 사례만 표시된다.
2. **Given** 홈페이지 초안과 게시본이 다를 때, **When** 익명 방문자가 홈페이지를 열면, **Then** 게시본만 표시된다.
3. **Given** 삭제된 사례의 경로를 알고 있을 때, **When** 익명 방문자가 직접 접근하면, **Then** 존재 여부를 구분할 수 없는 찾을 수 없음 화면을 본다.

---

### User Story 2 - 실제 고객 인증 (Priority: P1)

고객은 이메일과 비밀번호로 가입·로그인·로그아웃하고 비밀번호 재설정 안내를 받을 수 있다. 가입한 고객은 항상 고객 권한으로 시작한다.

**Why this priority**: 관리자 권한과 공개 데이터 경계를 신뢰하려면 실제 사용자 세션이 먼저 확립되어야 한다.

**Independent Test**: 신규 이메일로 가입하고 로그인한 뒤 자신의 계정만 확인하며 관리자 화면은 열 수 없는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 가입되지 않은 이메일일 때, **When** 고객이 유효한 비밀번호로 가입하면, **Then** 고객 프로필이 생성되고 관리자 권한은 부여되지 않는다.
2. **Given** 유효한 고객 계정일 때, **When** 로그인하면, **Then** 서버에서도 검증 가능한 세션이 생성된다.
3. **Given** 고객 세션일 때, **When** 관리자 주소에 접근하면, **Then** 관리자 기능이 노출되지 않고 로그인 또는 권한 부족 화면으로 이동한다.
4. **Given** 임의의 이메일일 때, **When** 비밀번호 재설정을 요청하면, **Then** 계정 존재 여부와 무관하게 같은 안내를 받는다.

---

### User Story 3 - 관리자 콘텐츠 운영 (Priority: P1)

활성 관리자는 사례, 태그, 영상, 가격과 홈페이지 콘텐츠를 만들고 수정하며 검토 후 게시·비공개 전환·삭제할 수 있다.

**Why this priority**: 관리자가 실제 사이트 내용을 변경하는 것이 본 프로젝트의 핵심 운영 목적이다.

**Independent Test**: 활성 관리자 세션으로 사례와 홈페이지 초안을 수정·게시하고 공개 화면에 즉시 반영되는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 활성 관리자일 때, **When** 비공개 사례를 작성하면, **Then** 저장되지만 공개 화면에는 나타나지 않는다.
2. **Given** 필수 개인정보 검토와 공개 미디어 메타데이터가 준비되었을 때, **When** 관리자가 사례를 게시하면, **Then** 전체 검증이 성공한 경우에만 공개된다.
3. **Given** 홈페이지 초안을 수정했을 때, **When** 게시하면, **Then** 이전 게시본이 버전으로 남고 공개 화면에는 새 게시본이 보인다.
4. **Given** 비활성 관리자일 때, **When** 관리자 변경을 시도하면, **Then** 모든 변경이 거부된다.

---

### User Story 4 - 슈퍼 관리자 권한 관리 (Priority: P2)

슈퍼 관리자는 기존 고객 계정에 관리자 권한을 부여하거나 회수하고 활성 상태를 변경할 수 있다. 일반 관리자는 이 기능을 사용할 수 없다.

**Why this priority**: 공개 회원가입과 관리자 권한 생성을 분리하여 계정 탈취와 권한 상승을 방지한다.

**Independent Test**: 일반 관리자와 슈퍼 관리자가 같은 역할 변경을 시도해 전자는 거부되고 후자는 감사 기록과 함께 성공하는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 일반 관리자일 때, **When** 다른 사용자의 역할 변경을 시도하면, **Then** 거부되고 역할은 유지된다.
2. **Given** 슈퍼 관리자일 때, **When** 기존 고객을 관리자로 승격하면, **Then** 역할이 변경되고 변경 전후 값과 사유가 감사 기록에 남는다.
3. **Given** 마지막 활성 슈퍼 관리자일 때, **When** 자신의 권한을 제거하거나 비활성화하려 하면, **Then** 작업이 거부된다.

---

### User Story 5 - Storage 지연 활성화 (Priority: P3)

Storage가 비활성인 동안 관리자는 실제 파일이 업로드되었다고 오인하지 않는다. 향후 계정 준비 후 운영 절차만으로 원본·공개 버킷을 활성화할 수 있다.

**Why this priority**: 데이터와 인증 연동을 막지 않으면서 비용과 운영 조건이 확정되지 않은 파일 저장을 안전하게 연기한다.

**Independent Test**: Storage 비활성 설정에서 업로드가 실패 폐쇄되고, 준비된 활성화 검증이 버킷·정책·환경 설정 누락을 탐지하는지 확인한다.

**Acceptance Scenarios**:

1. **Given** Storage가 비활성일 때, **When** 관리자가 파일을 선택하면, **Then** 원격 업로드를 시도하지 않고 연결 대기 상태를 표시한다.
2. **Given** Storage 활성화 준비를 수행할 때, **When** 버킷이나 정책이 빠져 있으면, **Then** 활성화 검증이 실패한다.

### Edge Cases

- 인증 사용자는 생성되었지만 프로필 트리거가 실패한 경우 관리자 권한을 얻지 못하고 복구 가능한 오류가 기록되어야 한다.
- 세션은 유효하지만 프로필이 비활성화된 경우 다음 관리자 요청부터 거부되어야 한다.
- 동시에 같은 사례를 게시하거나 역할을 변경해도 부분 반영이나 중복 감사 기록이 생기지 않아야 한다.
- 공개 사례의 대표 미디어가 비공개·미준비 상태로 변경되면 게시 작업이 거부되어야 한다.
- 손상된 홈페이지 초안은 게시되지 않고 기존 게시본이 유지되어야 한다.
- 원격 마이그레이션 이력이 로컬과 다르면 적용을 중단하고 차이를 보고해야 한다.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 고객, 관리자, 슈퍼 관리자 역할을 구분하고 공개 가입에는 고객 역할만 부여해야 한다.
- **FR-002**: 시스템은 익명 사용자에게 게시된 콘텐츠만 제공하고 비공개·삭제·초안 데이터를 모든 조회 경로에서 차단해야 한다.
- **FR-003**: 고객은 자신의 프로필만 조회할 수 있고 자신의 역할 또는 관리자 활성 상태를 변경할 수 없어야 한다.
- **FR-004**: 활성 관리자만 사례, 미디어 메타데이터, 영상, 태그, 가격과 홈페이지 콘텐츠를 변경할 수 있어야 한다.
- **FR-005**: 사례 게시 시 필수 필드, 개인정보 검토, 공개 준비 미디어와 유효한 대표 미디어를 원자적으로 검증해야 한다.
- **FR-006**: 홈페이지 게시는 초안 검증, 이전 게시본 버전 생성과 게시본 교체를 원자적으로 수행해야 한다.
- **FR-007**: 관리자 역할 부여·회수와 활성 상태 변경은 슈퍼 관리자만 수행해야 한다.
- **FR-008**: 모든 관리자 권한 변경은 실행자, 대상, 변경 전후 값, 사유와 시각을 수정 불가능한 기록으로 남겨야 한다.
- **FR-009**: 마지막 활성 슈퍼 관리자의 권한 제거 또는 비활성화를 방지해야 한다.
- **FR-010**: 관리자 화면은 서버에서 검증된 세션과 현재 역할·활성 상태가 없으면 접근을 차단해야 한다.
- **FR-011**: 인증 오류와 비밀번호 재설정 응답은 계정 존재 여부를 노출하지 않아야 한다.
- **FR-012**: 서비스 키·사례 경로·태그 키의 중복, 사례당 미디어 69개, 영상 3개와 대표 미디어 1개 제한을 신뢰 가능한 저장 경계에서 강제해야 한다.
- **FR-013**: 원격 데이터 변경은 버전 관리되는 변경 파일로 재현 가능해야 하며 적용 전 차이를 확인할 수 있어야 한다.
- **FR-014**: Storage는 기본 비활성 상태에서 원격 업로드를 시도하지 않아야 한다.
- **FR-015**: Storage 활성화 준비물은 원본 비공개와 검토 완료 공개 파일의 접근 경계를 정의해야 한다.
- **FR-016**: 초기 슈퍼 관리자, 관리자 비밀번호, 서비스 역할 비밀값 또는 데이터베이스 비밀번호를 소스·seed·브라우저 저장소에 포함하지 않아야 한다.
- **FR-017**: 기존 브라우저 목업 데이터는 검토 없이 운영 데이터로 자동 이전되지 않아야 한다.

### Key Entities

- **프로필**: 인증 사용자에 대응하며 역할과 관리자 활성 상태를 가진다.
- **관리자 역할 감사 기록**: 역할 또는 활성 상태 변경의 수정 불가능한 이력이다.
- **서비스**: 네 가지 고정 청소 분류와 공개 표시 정보를 가진다.
- **사례**: 서비스, 설명, 공개 상태, 개인정보 검토와 게시 시각을 가진다.
- **사례 미디어**: 사례에 연결된 이미지 메타데이터, 단계, 순서, 대표·공개·준비 상태를 가진다.
- **사례 영상**: 사례에 연결된 YouTube 정보와 공개 순서를 가진다.
- **태그와 사례 태그**: 서비스별 분류와 사례 연결을 나타낸다.
- **사이트 콘텐츠와 버전**: 홈페이지 초안, 게시본과 복원 가능한 게시 이력을 가진다.
- **가격 항목**: 서비스별 가격 표시, 조건, 노출과 순서를 가진다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 익명·고객·비활성 관리자·관리자·슈퍼 관리자 다섯 권한 시나리오에서 허용·거부 결과가 100% 요구사항과 일치한다.
- **SC-002**: 게시되지 않은 사례와 홈페이지 초안은 목록, 직접 주소, 검색 결과와 관련 콘텐츠에서 0건 노출된다.
- **SC-003**: 유효한 고객은 가입·로그인·로그아웃·재설정 요청의 주요 흐름을 2분 이내 완료할 수 있다.
- **SC-004**: 관리자의 유효한 사례 또는 홈페이지 게시 결과가 새 공개 조회에서 5초 이내 확인된다.
- **SC-005**: 권한 변경 100%가 실행자·대상·변경 전후 값·사유·시각을 포함한 감사 기록을 만든다.
- **SC-006**: 실패한 게시·역할 변경에서는 부분 데이터 변경이 0건이다.
- **SC-007**: Storage 비활성 상태의 파일 선택 100%가 원격 업로드 없이 연결 대기 상태로 종료된다.

## Assumptions

- Supabase 프로젝트 연결 정보와 공개 클라이언트 환경변수는 이미 준비되어 있다.
- 이메일·비밀번호 인증을 사용하며 운영 전 이메일 리디렉션 주소와 SMTP를 별도로 검토한다.
- 네 개 서비스 키는 기존 PRD 값을 유지한다.
- 초기 슈퍼 관리자는 구현 완료 후 별도 운영 절차로 생성하며 이번 작업에서는 만들지 않는다.
- 실제 Storage 버킷 생성과 파일 업로드는 계정 플랜 확인 후 별도 승인으로 활성화한다.
- 기존 목업 저장소는 테스트 fixture와 개발 복구 참고용으로만 남길 수 있다.
