# Feature Specification: 고객·관리자 인증 경계

**Feature Branch**: `not-applicable-no-git-repository`

**Created**: 2026-07-23

**Status**: Awaiting written-spec review

**Input**: 고객 회원가입·로그인과 관리자 로그인을 제공한다. 관리자는 공개 회원가입할 수 없고, 향후 배포 단계에서 생성할 초기 슈퍼 관리자가 기존 계정에 관리자 권한을 부여·회수한다. 현재 슈퍼 계정은 생성하지 않으며 향후 Supabase 연동과 서버 권한 검증을 고려한다.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 고객 회원가입과 로그인 (Priority: P1)

고객은 이메일과 비밀번호로 계정을 만들고 로그인·로그아웃할 수 있다.

**Why this priority**: 고객 인증 기능의 기본 진입점이며 이후 고객별 기능의 전제가 된다.

**Independent Test**: 회원가입 화면에서 유효한 정보를 입력해 고객 세션을 만들고, 로그아웃 후 동일 계정으로 로그인하는 흐름을 검증한다.

**Acceptance Scenarios**:

1. **Given** 비회원 고객이 유효한 이메일과 비밀번호를 입력했을 때, **When** 가입을 제출하면, **Then** 고객 역할 계정 생성 결과와 다음 행동을 확인한다.
2. **Given** 유효한 고객 계정이 있을 때, **When** 올바른 정보로 로그인하면, **Then** 고객 계정 화면으로 이동한다.
3. **Given** 로그인된 고객이 로그아웃하면, **Then** 보호된 고객 화면 접근이 종료되고 공개 화면으로 돌아간다.

---

### User Story 2 - 고객 비밀번호 재설정 (Priority: P2)

고객은 비밀번호를 잊었을 때 이메일로 안전한 재설정 절차를 시작한다.

**Independent Test**: 등록 여부와 관계없이 같은 완료 문구가 표시되고 재설정 요청을 반복 제출하지 못하는지 검증한다.

**Acceptance Scenarios**:

1. **Given** 고객이 이메일을 제출했을 때, **When** 재설정을 요청하면, **Then** 계정 존재 여부를 노출하지 않는 동일한 안내를 받는다.
2. **Given** 요청이 처리 중일 때, **When** 다시 제출하려 하면, **Then** 중복 요청이 차단된다.

---

### User Story 3 - 관리자 로그인과 접근 차단 (Priority: P1)

관리자는 별도 관리자 로그인 화면을 사용하며, 관리자 역할이 없는 사용자는 관리자 화면에 접근하지 못한다.

**Independent Test**: 미인증·고객·관리자 상태별로 관리자 경로 접근 결과를 검증하고 관리자 회원가입 진입점이 없는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 미인증 사용자가 관리자 경로를 열었을 때, **Then** 관리자 로그인으로 이동한다.
2. **Given** 고객 역할 사용자가 관리자 로그인을 시도했을 때, **Then** 계정 존재 또는 역할 세부정보를 노출하지 않는 접근 거부 안내를 받는다.
3. **Given** 관리자 로그인 화면에서, **Then** 회원가입 링크·가입 폼·관리자 계정 생성 동작이 존재하지 않는다.
4. **Given** 현재 슈퍼 계정과 관리자 계정이 설정되지 않은 목업 상태에서, **When** 로그인을 시도하면, **Then** 운영 연동 전에는 관리자 로그인이 활성화되지 않았다는 안전한 안내를 받는다.

---

### User Story 4 - 슈퍼 관리자의 관리자 권한 관리 (Priority: P3)

향후 슈퍼 관리자는 기존 고객 계정의 관리자 권한을 부여하거나 회수할 수 있다.

**Independent Test**: 슈퍼 관리자, 일반 관리자, 고객 역할별로 권한 관리 화면과 역할 변경 동작의 허용 여부를 검증한다.

**Acceptance Scenarios**:

1. **Given** 슈퍼 관리자가 기존 고객 계정을 선택했을 때, **When** 관리자 권한 부여를 확인하면, **Then** 대상 계정은 활성 관리자 역할을 갖는다.
2. **Given** 일반 관리자 또는 고객이 권한 관리 화면에 접근할 때, **Then** 사용자 목록과 역할 변경 동작을 볼 수 없다.
3. **Given** 슈퍼 관리자가 자신의 마지막 슈퍼 관리자 권한을 제거하려 할 때, **Then** 동작이 차단된다.
4. **Given** Supabase가 연결되지 않은 현재 단계에서, **When** 권한 관리 화면을 열면, **Then** 잠금 상태와 필요한 연동 조건만 표시된다.

### Edge Cases

- 이메일 앞뒤 공백과 대소문자 차이는 정규화한다.
- 비밀번호 정책 미충족은 요구 조건을 구체적으로 안내하되 입력한 비밀번호를 기록하지 않는다.
- 연속 실패, 만료 세션, 네트워크 실패와 중복 제출에 복구 행동을 제공한다.
- 로그인 오류는 이메일 존재 여부와 역할 여부를 구분해 노출하지 않는다.
- 비활성 관리자, 권한 회수된 관리자와 유효하지 않은 세션은 관리자 화면에서 즉시 제외한다.
- 관리자 권한 변경 중 실패하면 기존 역할을 유지한다.
- 슈퍼 관리자 계정이 아직 없을 때 자동 생성하거나 기본 비밀번호로 대체하지 않는다.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 고객은 이메일과 비밀번호로 회원가입할 수 있어야 한다.
- **FR-002**: 고객은 로그인, 로그아웃과 비밀번호 재설정 요청을 수행할 수 있어야 한다.
- **FR-003**: 회원가입·로그인 입력은 이메일 형식, 비밀번호 정책과 필수 동의를 검증해야 한다.
- **FR-004**: 비밀번호와 인증 토큰은 화면, 로그, URL과 일반 콘텐츠 저장소에 노출되지 않아야 한다.
- **FR-005**: 관리자 로그인은 고객 로그인과 분리된 경로와 안내를 제공해야 한다.
- **FR-006**: 관리자 회원가입 화면, 링크와 공개 계정 생성 동작은 존재하지 않아야 한다.
- **FR-007**: 관리자 화면은 인증된 활성 `admin` 또는 `super_admin`만 접근할 수 있어야 한다.
- **FR-008**: 고객 역할과 미인증 사용자는 관리자 데이터와 관리자 화면을 볼 수 없어야 한다.
- **FR-009**: 역할은 `customer`, `admin`, `super_admin`으로 구분해야 한다.
- **FR-010**: 관리자 역할 부여·회수는 인증된 `super_admin`만 수행할 수 있어야 한다.
- **FR-011**: 초기 슈퍼 관리자 계정은 현재 코드, seed, 브라우저 저장소 또는 공개 설정에 생성되지 않아야 한다.
- **FR-012**: 운영 연동 전 관리자 로그인은 성공 세션을 위조하지 않고 미설정 상태를 안내해야 한다.
- **FR-013**: 권한 관리 화면은 운영 연동 전 잠금 상태를 제공하고 임의 역할 변경을 허용하지 않아야 한다.
- **FR-014**: 마지막 활성 슈퍼 관리자의 슈퍼 권한 제거는 차단해야 한다.
- **FR-015**: 로그인 실패, 권한 거부와 역할 변경은 운영 단계에서 감사 가능한 사건으로 기록되어야 한다.
- **FR-016**: 인증 오류는 계정 존재 여부나 구체적인 역할 여부를 추측할 수 없게 표현해야 한다.
- **FR-017**: 중복 제출을 방지하고 만료·실패 상태에 재시도 또는 다시 로그인 행동을 제공해야 한다.
- **FR-018**: 고객·관리자 인증 화면은 390px와 1440px에서 기존 사이트의 입력, 버튼, 색상과 초점 스타일을 재사용해야 한다.
- **FR-019**: 실제 운영 권한은 클라이언트가 전달한 역할값이 아니라 신뢰 가능한 세션과 저장소 정책으로 검증되어야 한다.
- **FR-020**: 인증 구현은 목업 인증 어댑터와 향후 운영 인증 어댑터를 UI 변경 없이 교체할 수 있어야 한다.

### Key Entities

- **사용자 계정**: 이메일, 인증 상태와 생성 시각을 가진 고객 또는 운영 사용자다.
- **프로필**: 표시 이름, 활성 상태와 고객용 기본 정보를 가진다.
- **역할 할당**: 사용자와 역할의 관계, 부여·회수 주체와 시각을 기록한다.
- **인증 세션**: 사용자, 만료 상태와 신뢰 가능한 역할 확인 결과를 나타낸다.
- **권한 변경 기록**: 역할 변경 대상, 이전·새 역할, 수행자와 결과를 나타낸다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 고객은 모바일과 데스크톱에서 회원가입 또는 로그인을 2분 이내에 완료할 수 있다.
- **SC-002**: 관리자 회원가입 경로, 링크와 공개 관리자 계정 생성 성공 건수는 0건이다.
- **SC-003**: 미인증 사용자와 고객 역할의 관리자 화면·데이터 접근 성공 건수는 0건이다.
- **SC-004**: 슈퍼 관리자 이외 역할의 관리자 권한 변경 성공 건수는 0건이다.
- **SC-005**: 코드, 브라우저 저장소와 공개 빌드 산출물에서 초기 슈퍼 관리자 비밀번호 또는 인증 비밀 노출 건수는 0건이다.
- **SC-006**: 핵심 인증 흐름은 키보드만으로 완료 가능하고 모든 오류가 해당 입력 또는 상태 영역과 연결된다.
- **SC-007**: 비기술 사용자 대상 확인에서 90% 이상이 고객 로그인과 관리자 로그인의 목적을 첫 화면에서 구분한다.

## Assumptions

- 고객 계정은 향후 고객별 기능을 위한 기반이며 이번 기능에서 결제·예약·리뷰를 추가하지 않는다.
- 고객 인증은 이메일과 비밀번호를 기본으로 하고 소셜 로그인은 후속 범위로 둔다.
- 관리자 계정은 고객 계정 생성 후 슈퍼 관리자가 권한을 부여하는 방식으로 만든다.
- 현재 목업에는 실제 관리자 또는 슈퍼 관리자 자격 증명을 포함하지 않는다.
- 실제 운영 연동 전까지 관리자 인증 성공과 권한 변경은 활성화하지 않는다.
- 향후 Supabase 연동 단계에서 서버 세션 검사, 저장소 권한 정책, 요청 제한과 감사 기록을 완성한다.
