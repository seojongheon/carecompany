# 1. 제품 개요

## 1.1 제품 한 줄 설명

위생의 기술이 수행한 화장실, 에어컨, 아파트 유리창, 상가 유리창 청소 현장을 고객이 사진 중심으로 쉽게 탐색하고, 관리자가 휴대전화로 현장 사진을 체계적으로 등록·공개할 수 있는 SEO 친화적 포트폴리오 웹사이트다.

## 1.2 제품의 본질

이 제품은 단순 사진 보관함이나 기업 소개 페이지가 아니다.

다음 세 가지 역할을 동시에 수행한다.

1. **고객용 영업 포트폴리오**
   - 고객이 자신의 상황과 비슷한 청소 사례를 찾는다.
   - 작업 전후와 과정을 통해 업체의 실력을 판단한다.
   - 가격 정책과 작업 방식을 이해한다.

2. **검색 유입용 콘텐츠 사이트**
   - 서비스별 독립 페이지와 현장별 독립 상세페이지를 제공한다.
   - 천안·아산 지역과 실제 청소 사례를 중심으로 검색엔진이 이해할 수 있는 콘텐츠를 만든다.

3. **관리자용 포트폴리오 관리 도구**
   - 현장에서 휴대전화로 여러 사진을 업로드한다.
   - 사진을 작업 전·과정·후로 분류한다.
   - 공개 여부와 개인정보 검토 상태를 관리한다.
   - 공개한 사례만 고객 화면과 검색엔진에 노출한다.

# 2. 배경과 문제 정의

## 2.1 현재 문제

청소업체의 작업 자료는 일반적으로 휴대전화 사진첩, 메신저, 블로그, SNS 등에 흩어져 있다. 이 방식은 다음 문제를 만든다.

- 현장별 사진이 섞여 나중에 찾기 어렵다.
- 고객에게 비슷한 사례를 빠르게 보여주기 어렵다.
- 인스타그램식 사진 나열만으로는 장소, 오염 문제, 작업 범위가 명확하지 않다.
- 사진이 검색 가능한 독립 콘텐츠로 축적되지 않는다.
- 공개하면 안 되는 고객 개인정보가 섞일 위험이 있다.
- 자료가 많아질수록 게시와 관리가 번거로워진다.
- 긴 서비스 페이지에서 원하는 사례를 보려면 반복적인 페이지 이동이 필요하다.

## 2.2 해결 방향

- 홈에서 네 가지 서비스를 바로 선택하게 한다.
- 각 서비스 페이지에서 해당 서비스의 여러 현장 사례를 사진 그리드로 탐색하게 한다.
- 현장 카드를 누르면 현재 페이지 문맥을 유지한 빠른보기 모달을 연다.
- 모달에서 전체 상세페이지로 이동할 수 있게 한다.
- 직접 URL, 검색 결과, 공유 링크로 들어오면 모달이 아닌 독립 상세페이지를 제공한다.
- 관리자는 휴대전화로 한 번에 최대 20장, 현장당 최대 69장의 사진을 등록한다.
- 비공개 사례와 비공개 사진은 고객에게 존재하지 않는 것처럼 처리한다.
- 영상은 직접 업로드하지 않고 YouTube 링크만 등록한다.

# 3. 제품 목표

## 3.1 핵심 목표

### G-01. 고객이 필요한 서비스를 즉시 찾는다

홈 첫 화면과 주요 내비게이션에서 다음 네 서비스를 직접 선택할 수 있어야 한다.

- 화장실 청소
- 에어컨 청소
- 아파트 유리창 청소
- 상가 유리창 청소

### G-02. 고객이 여러 실제 현장을 부담 없이 탐색한다

서비스 페이지에 현장 사례가 충분히 많다는 전제에서, 고객이 사진을 빠르게 훑고 관심 사례를 모달로 확인할 수 있어야 한다.

### G-03. 한 현장의 작업 품질을 구체적으로 설명한다

개별 상세페이지에서 작업 전 상태, 작업 범위, 과정, 결과, 사진, YouTube 영상을 함께 제공한다.

### G-04. 검색 가능한 사례 자산을 지속적으로 축적한다

모든 공개 사례는 고유 URL, 제목, 설명, 대표 이미지 및 구조화 가능한 데이터를 가진다.

### G-05. 관리자가 휴대전화로 작업 자료를 빠르게 저장한다

현장에서 최소한의 정보와 사진을 먼저 저장하고, 이후 모바일이나 데스크톱에서 설명과 공개 설정을 마무리할 수 있어야 한다.

### G-06. 비공개 자료를 고객에게 완전히 숨긴다

비공개 사례, 비공개 사진, 문의 첨부파일 및 내부 원본은 공개 쿼리, 화면, 사이트맵과 직접 URL에서 노출되지 않아야 한다.

## 3.2 MVP 성공 기준

정식 배포 전 아래 기준을 충족해야 한다.

- 홈에서 네 서비스 상세페이지까지 1회 클릭으로 이동 가능하다.
- 각 서비스 페이지에서 공개 사례가 사진 그리드로 표시된다.
- 사례 카드 클릭 시 빠른보기 모달이 열린다.
- 모달의 뒤로가기, 닫기, 이전·다음 사례 탐색이 정상 동작한다.
- 같은 사례 URL을 직접 열면 독립 상세페이지가 나온다.
- 필터와 스크롤 위치가 모달을 닫은 뒤 유지된다.
- 공개 사례만 고객 화면과 사이트맵에 포함된다.
- 관리자가 모바일에서 여러 사진을 업로드하고 공개할 수 있다.
- 한 번의 사진 선택은 최대 20장, 한 사례의 전체 사진은 최대 69장으로 제한된다.
- 잘못된 YouTube 링크는 게시 전에 검증된다.
- 핵심 고객 페이지가 모바일과 데스크톱에서 사용 가능하다.
- 페이지의 핵심 콘텐츠는 JavaScript 실행 전에도 검색엔진이 읽을 수 있는 HTML로 제공된다.
- 주요 E2E 사용자 흐름 테스트가 통과한다.

# 5. 확정된 제품 결정

## 5.1 브랜드

- 브랜드명은 `위생의 기술`이다.
- 기존 파란색 계열의 아이코닉한 로고가 있으나 MVP 제작 중 사용자가 직접 교체한다.
- Figma와 초기 구현에서는 가변 너비 로고 플레이스홀더를 사용한다.
- 파란색 계열의 메인 색상을 사용한다.
- 파란색과 충돌하는 강한 원색 조합은 피한다.
- 흰색, 밝은 회색, 차콜을 기본 중립색으로 사용한다.
- 디자인의 핵심은 장식이 아니라 실제 작업 사진, 청결함, 신뢰감이다.

## 5.2 서비스 분류

홈에는 다음 네 카드를 독립적으로 노출한다.

1. 화장실 청소
2. 에어컨 청소
3. 아파트 유리창 청소
4. 상가 유리창 청소

정보 구조상 유리창 청소를 상위 분류로 묶을 수 있으나, 홈에서는 고객이 한 단계 더 선택하지 않도록 두 서비스를 독립 카드로 제공한다.

## 5.3 지역

- MVP의 서비스 문구와 SEO는 천안·아산 중심이다.
- 데이터 구조는 향후 지역 확장이 가능해야 한다.
- 실제 사례가 없는 타 지역 전용 페이지를 생성하지 않는다.

## 5.4 가격

- 개별 현장 사례에 당시 작업 가격을 표시하지 않는다.
- 별도의 가격 안내 페이지를 제공한다.
- 각 서비스 페이지에는 해당 서비스 가격의 짧은 요약과 전체 가격표 링크를 제공한다.
- 실제 가격 데이터와 문구는 관리자가 수정·추가·삭제·정렬하고 게시할 수 있는 콘텐츠 영역으로 만든다.
- 가격 항목은 개별 사례의 당시 작업비와 연결하지 않으며, 서비스별 기준·범위·포함 사항·변동 조건을 안내한다.

## 5.5 견적 플로팅 버튼

- 고객 화면 우측 하단에 원형 `+` 플로팅 버튼을 둔다.
- 이 버튼은 향후 사진 첨부 견적 폼을 여는 진입점이다.
- MVP에서는 버튼 컴포넌트와 시각 상태만 구현한다.
- 견적 폼, 사진 첨부, 문의 저장 기능은 이번 MVP 범위에서 제외한다.
- 버튼은 접근 가능한 이름으로 `사진 견적 열기` 의미를 가진다.
- 빠른보기 모달과 사진 라이트박스가 열린 동안 버튼은 숨긴다.

## 5.6 영상

- 영상 파일은 직접 업로드하지 않는다.
- 관리자는 YouTube URL만 입력한다.
- 사례당 최대 3개의 YouTube 링크를 등록할 수 있다.
- 고객이 재생하기 전에는 플레이어 전체를 로딩하지 않고 썸네일 기반의 지연 로딩 UI를 사용한다.

## 5.7 공개 상태

- 새로운 포트폴리오 사례의 기본 상태는 `비공개`다.
- 관리자가 공개 전 체크리스트를 통과한 뒤 `공개`로 변경한다.
- 비공개 사례는 고객에게 슬롯, 잠금, 비공개 문구, 개수 등 어떤 흔적도 보여주지 않는다.
- 비공개 사례 URL은 고객에게 일반 404로 처리한다.

# 6. 사용자 정의

## 6.1 고객 사용자

### 유형 A: 서비스 탐색 고객

홈에서 필요한 서비스를 선택하고 실제 사례를 빠르게 확인하려는 고객이다.

### 유형 B: 검색 유입 고객

`천안 화장실 청소`, `아산 아파트 유리창 청소` 또는 특정 문제를 검색해 서비스나 개별 사례 페이지로 직접 들어온 고객이다.

### 유형 C: 비교 검토 고객

업체에 문의하기 전 여러 사례, 작업 과정, 가격 정책과 업체 정보를 비교하려는 고객이다.

## 6.2 관리자

- 위생의 기술 대표자 또는 내부 운영자
- 주요 관리 기기는 휴대전화
- 사진 업로드는 현장에서 수행할 가능성이 높다.
- 상세 설명 수정과 사진 정리는 모바일과 데스크톱 모두에서 수행한다.
- MVP에서는 관리자 권한을 한 종류로만 취급한다.

# 7. 핵심 사용자 여정

## 7.1 홈에서 서비스와 사례 탐색

1. 고객이 홈에 진입한다.
2. 첫 화면에서 업체의 서비스와 천안·아산 중심 활동을 이해한다.
3. 네 개의 서비스 카드 중 하나를 선택한다.
4. 서비스 상세페이지에서 대표 결과와 여러 현장 사례를 확인한다.
5. 사례 카드를 선택한다.
6. 빠른보기 모달에서 주요 사진과 요약을 확인한다.
7. 다음 사례를 계속 보거나 전체 사례 상세페이지로 이동한다.
8. 가격 안내 또는 향후 견적 버튼을 확인한다.

## 7.2 검색 결과에서 개별 사례로 직접 진입

1. 검색 결과 또는 공유 링크에서 개별 사례 URL을 연다.
2. 전체 상세페이지가 표시된다.
3. 작업 전·과정·후 사진과 현장 설명을 읽는다.
4. 관련 사례를 통해 같은 서비스의 다른 현장을 확인한다.
5. 서비스 페이지 또는 가격 안내로 이동한다.

## 7.3 서비스 페이지에서 빠른보기 모달 탐색

1. 사용자가 서비스 필터를 선택한다.
2. 특정 사례 카드를 클릭한다.
3. URL이 해당 사례 URL로 변경되며 모달이 열린다.
4. 이전·다음 사례를 탐색한다.
5. 브라우저 뒤로가기를 누르면 페이지를 떠나는 대신 모달이 닫힌다.
6. 기존 필터와 스크롤 위치가 유지된다.

## 7.4 모바일 관리자 현장 등록

1. 관리자가 로그인한다.
2. `새 포트폴리오`를 선택한다.
3. 서비스, 작업 지역, 임시 제목 등 최소 정보를 입력한다.
4. 휴대전화에서 사진을 최대 20장 선택한다.
5. 업로드 진행률을 확인한다.
6. 필요하면 여러 번 반복하여 총 69장 미만으로 추가한다.
7. 완료된 사진과 입력 정보는 즉시 비공개 초안으로 저장된다.
8. 관리자는 나중에 사진 분류, 순서, 설명, YouTube 링크를 정리한다.
9. 공개 전 개인정보 체크리스트를 확인한다.
10. 사례를 공개한다.

## 7.5 사례 비공개 전환

1. 관리자가 공개 사례를 연다.
2. 상태를 비공개로 변경한다.
3. 고객용 목록, 관련 사례, 사이트맵 및 직접 URL에서 즉시 제거한다.
4. 고객은 해당 사례가 존재하는지 알 수 없다.
5. 관리자 화면에서는 계속 편집할 수 있다.

# 9. UX 원칙

## UX-01. 사진이 주인공이어야 한다

사진 위에 긴 문장을 겹치거나 과도한 장식과 애니메이션을 사용하지 않는다.

## UX-02. 고객은 한 단계 덜 클릭해야 한다

홈에서 유리창 청소를 누른 뒤 아파트와 상가를 다시 고르게 하지 않는다.

## UX-03. 빠른 탐색과 깊은 설명을 분리한다

- 빠른보기 모달: 관심 여부를 판단하는 핵심 사진과 짧은 정보
- 상세페이지: 전체 작업 내용, 사진, 영상, SEO 콘텐츠

## UX-04. 모달은 페이지 문맥을 보존해야 한다

필터, 스크롤 위치, 보고 있던 사례 순서를 유지한다.

## UX-05. 고객에게 내부 관리 상태를 노출하지 않는다

비공개 사례나 사진은 빈 슬롯, 잠금 상태, 누락 번호 등으로 표시하지 않는다.

## UX-06. 관리자 작업은 중단되어도 보존돼야 한다

완료된 업로드와 기본 정보는 자동으로 저장하며, 다시 접속해 이어서 편집할 수 있어야 한다.

## UX-07. 모바일과 데스크톱의 기능은 같고 표현은 달라야 한다

데스크톱 UI를 단순 축소하지 않는다. 모바일에서는 전체 화면 모달, 하단 고정 작업 버튼, 터치 중심 조작을 사용한다.

## UX-08. 필수 행동만 강조한다

한 화면에 여러 CTA를 경쟁시키지 않는다. `+` 버튼은 향후 견적 기능의 일관된 단일 진입점으로 유지한다.

# 10. 디자인 방향

## 10.1 시각적 성격

- 깨끗함
- 정돈됨
- 실제 현장감
- 전문적이지만 과도하게 기업적이지 않음
- 신뢰감
- 사진 중심
- 모바일 친화적

## 10.2 색상 원칙

정확한 HEX 값은 Figma 단계에서 확정하되 다음 역할을 지킨다.

- **Primary:** 기존 로고와 어울리는 중간 또는 진한 파란색
- **Primary light:** 선택 상태, 배경 강조, 정보 박스용 연한 파란색
- **Background:** 흰색 또는 매우 밝은 회색
- **Surface:** 카드와 모달용 흰색
- **Text primary:** 차콜 또는 거의 검은색
- **Text secondary:** 중간 회색
- **Border:** 밝은 회색
- **Success:** 공개 완료 상태에만 제한적으로 사용
- **Danger:** 삭제·오류에만 사용

서비스별로 서로 다른 강한 고유 색상을 부여하지 않는다. 카테고리는 텍스트와 사진, 작은 라벨로 구분한다.

## 10.3 타이포그래피

- 한글 가독성이 좋은 산세리프 글꼴
- 사용자 기기와 웹 성능을 고려한 폰트 로딩
- 본문 최소 16px
- 모바일에서 지나치게 작은 보조 텍스트 금지
- 제목, 본문, 캡션, 라벨의 계층을 명확하게 구분

## 10.4 이미지 표현

- 카드 썸네일 비율을 통일한다.
- 전후 사진에는 텍스트 라벨 `청소 전`, `청소 후`를 사용한다.
- 색상만으로 전후를 구분하지 않는다.
- 사진 위 텍스트 오버레이는 최소화한다.
- 사진 확대 시 캡션과 현재 번호를 제공한다.
- 작업 전후 사진은 가능하면 비슷한 촬영 각도를 사용한다.

## 10.5 모션

- 카드 호버, 탭 전환, 모달 열림 등 짧고 절제된 전환만 사용한다.
- 자동 재생 슬라이드와 배경 영상은 사용하지 않는다.
- `prefers-reduced-motion` 설정을 존중한다.

# 11. 반응형 기준

구현 프레임워크의 기본 중단점을 활용하되, 콘텐츠 기반으로 최종 조정한다.

## 11.1 모바일

- 기준 폭: 약 360px 이상
- 서비스 카드: 2열
- 사례 카드: 기본 2열
- 빠른보기: 전체 화면 모달
- 사진 썸네일: 가로 스크롤
- 관리자 편집: 한 열 단계형 폼
- 주요 저장·게시 버튼: 하단 고정 영역 사용 가능
- 터치 대상 최소 크기를 확보한다.

## 11.2 태블릿

- 서비스 카드: 2열 또는 4열
- 사례 카드: 2~3열
- 모달: 화면 크기에 따라 전체 화면 또는 중앙 모달
- 관리자 편집: 한 열 또는 제한적 2열

## 11.3 데스크톱

- 최대 콘텐츠 폭을 제한하여 지나치게 늘어나지 않게 한다.
- 서비스 카드: 4열
- 사례 카드: 3열, 넓은 화면에서 최대 4열
- 빠른보기 모달: 좌측 사진 60~65%, 우측 정보 35~40%
- 관리자 편집: 본문 편집과 미리보기 또는 설정 패널을 2열로 제공 가능

## 11.4 공통 원칙

- 가로 스크롤이 발생하지 않아야 한다.
- 사진의 레이아웃 이동을 방지하도록 크기 정보를 확보한다.
- 모바일과 데스크톱 모두 동일한 콘텐츠와 공개 규칙을 적용한다.

# 12. 고객용 페이지 요구사항

## 12.1 홈 페이지

### 목적

업체가 무엇을 하는지 즉시 이해시키고 네 서비스 중 하나로 빠르게 이동시킨다.

### 섹션 순서

1. 글로벌 헤더
2. 히어로
3. 네 서비스 선택 카드
4. 대표 전후 사례
5. 최근 또는 추천 사례
6. 작업 방식 요약
7. 가격 안내 진입 영역
8. 신뢰 요소
9. 다른 페이지 진입 CTA
10. 푸터
11. 플로팅 `+` 버튼

### 히어로 요구사항

- 실제 작업 사진 또는 대표 작업 결과 이미지를 사용한다.
- 브랜드 한 줄 메시지
- 천안·아산 중심 서비스 문구
- `작업 사례 보기` 링크
- 로고 플레이스홀더
- 지나치게 큰 높이로 사례 접근을 늦추지 않는다.

### 서비스 카드 요구사항

각 카드에는 다음 정보를 포함한다.

- 서비스명
- 대표 이미지
- 한 줄 설명
- 링크 전체 클릭 영역
- 키보드 포커스 상태
- 모바일 2×2 배치

### 대표 사례

- 최대 4개
- 전후 차이가 분명한 사례를 우선한다.
- 특정 서비스만 과도하게 편중하지 않는다.
- 카드 클릭 시 해당 사례 상세 또는 빠른보기 흐름으로 연결한다.

네 서비스는 동일한 템플릿을 공유하고 콘텐츠만 달라진다.

### 페이지 목적

- 해당 서비스의 범위와 장점을 설명한다.
- 많은 현장 사례를 사진 중심으로 탐색하게 한다.
- 작업 과정, 가격, FAQ로 불안을 해소한다.

### 공통 섹션 순서

1. 헤더
2. 서비스 히어로
3. 페이지 내부 스티키 탭
4. 서비스 요약
5. 대표 전후 사례
6. 작업 범위
7. 사례 필터
8. 사례 그리드
9. `사례 더 보기`
10. 작업 과정
11. 가격 요약
12. FAQ
13. 다른 서비스
14. 푸터
15. 플로팅 `+` 버튼

### 스티키 탭

항목 예시:

- 서비스 소개
- 작업 범위
- 시공 사례
- 작업 과정
- 가격
- FAQ

데스크톱에서는 헤더 아래 가로 탭, 모바일에서는 가로 스크롤 탭으로 제공한다. 현재 보고 있는 섹션을 시각적으로 표시한다.

### 사례 필터

MVP에서는 검색창을 제공하지 않는다.

필터 상태는 URL 검색 파라미터에 반영하여 새로고침과 뒤로가기에 유지한다.

공통 필터:

- 전체
- 공간 유형
- 주요 작업·오염 태그

서비스별 초기 필터 예:

#### 화장실 청소

- 주거지
- 상가·공장
- 물때
- 곰팡이
- 요석
- 석회
- 니코틴
- 샤워부스
- 변기
- 타일·줄눈

#### 에어컨 청소

- 벽걸이형

- 시스템형
- 필터
- 내부 오염

### 사례 그리드

- 공개 사례만 표시한다.
- 기본 정렬은 `대표 우선 → 공개일 최신순`이다.
- 모바일 최초 10~12개
- 태블릿 최초 12개
- 데스크톱 최초 12~16개
- 이후 명시적 `사례 더 보기` 버튼으로 추가 로딩한다.
- 무한 스크롤은 사용하지 않는다.
- 로딩 중 스켈레톤을 표시하되 레이아웃이 이동하지 않아야 한다.
- 결과가 없으면 비공개 슬롯을 보여주지 않고 선택 필터에 공개 사례가 없다는 일반 메시지를 표시한다.

### 사례 카드 내용

- 대표 썸네일
- 선택적으로 전후 분할 썸네일
- 서비스 라벨
- 사례 제목
- 지역
- 공간 유형
- 공개 사진 수 또는 영상 여부 아이콘은 선택적으로 표시
- 카드 전체 클릭 가능
- hover/focus 상태
- 카드에서 실제 작업 가격은 표시하지 않는다.

## 12.3 전체 사례 페이지

### 목적

네 서비스의 모든 공개 사례를 한 화면에서 탐색한다.

### 필터

- 전체
- 화장실
- 에어컨
- 아파트 유리창
- 상가 유리창

선택적으로 두 번째 줄에 공간 유형 또는 주요 태그 필터를 제공한다. 복잡한 조합 필터는 MVP에서 제외한다.

### 표시 규칙

- 공개 사례만 표시한다.
- 동일한 사례 카드 컴포넌트를 재사용한다.
- 서비스 페이지와 같은 빠른보기 모달 흐름을 사용한다.
- 필터는 URL 상태로 유지한다.
- 페이지당 초기 데이터와 더 보기 방식은 서비스 페이지와 동일하다.

## 12.4 사례 빠른보기 모달

### 목적

서비스 또는 전체 사례 목록을 떠나지 않고 한 현장의 핵심 결과를 빠르게 판단하게 한다.

### 라우팅

- 사례 카드의 소프트 내비게이션은 해당 사례의 canonical URL로 이동하되 현재 목록 위에 모달로 표시한다.
- 브라우저 주소는 `/portfolio/[slug]`를 반영한다.
- 같은 URL을 새로고침하거나 직접 열면 모달이 아닌 전체 상세페이지를 표시한다.
- 브라우저 뒤로가기는 먼저 모달을 닫는다.
- 앞으로가기는 모달을 다시 열 수 있어야 한다.
- `전체 사례 보기`는 인터셉트되지 않은 전체 상세페이지로 이동한다.

### 데스크톱 레이아웃

- 중앙 대형 모달
- 좌측: 대표 이미지와 선택된 핵심 이미지
- 우측: 제목, 지역, 공간 유형, 주요 오염, 작업 범위, 짧은 설명, 버튼
- 이전·다음 사례 탐색
- 현재 위치 예: `4 / 27`
- 닫기 버튼

### 모바일 레이아웃

- 전체 화면
- 상단 고정 뒤로가기/닫기
- 큰 대표 이미지
- 좌우 스와이프 이미지 탐색
- 핵심 정보
- 전체 사례 보기
- 이전·다음 사례 탐색
- 본문 뒤의 플로팅 `+` 버튼은 숨김

### 콘텐츠 제한

모달은 다음만 제공한다.

- 대표 전후 사진
- 추가 핵심 사진을 합해 최대 5~7장
- 사례 제목
- 지역 및 공간
- 주요 오염
- 작업 범위
- 짧은 설명 2~3문장
- 전체 공개 사진 수
- YouTube 영상 존재 여부
- 전체 사례 보기 링크

모달 안에 전체 작업 설명, 모든 사진, 전체 영상 플레이어와 긴 FAQ를 넣지 않는다.

### 모달 UX 요구사항

- Escape로 닫기
- 포커스 트랩
- 닫은 뒤 원래 카드로 포커스 복귀
- 배경 스크롤 잠금
- 필터 및 스크롤 위치 유지
- 키보드 좌우 방향키로 이전·다음 이동
- 모바일 스와이프 지원
- 모달 중첩 금지
- 모달 내부 사진 클릭으로 또 다른 라이트박스를 열지 않는다.

## 12.5 개별 사례 상세페이지

### 목적

한 현장의 문제, 작업, 결과를 충분히 설명하며 공유와 검색 유입을 담당한다.

### 섹션 순서

1. 헤더
2. Breadcrumb
3. 사례 제목
4. 작업 요약 정보
5. 대표 전후 비교
6. 현장 상태 설명
7. 작업 범위
8. 작업 과정
9. 작업 전 갤러리
10. 작업 과정 갤러리
11. 작업 후 갤러리
12. 세부 사진
13. YouTube 영상
14. 결과 요약
15. 관련 사례
16. 서비스 페이지 링크
17. 가격 안내 링크
18. 푸터
19. 플로팅 `+` 버튼

비어 있는 섹션은 제목이나 공간을 남기지 않고 완전히 숨긴다.

### 작업 요약 정보

- 서비스
- 표시 지역
- 공간 유형
- 주요 오염
- 작업 범위
- 작업 시기
- 공개 가능한 경우에만 작업 소요 시간
- 정확한 주소, 고객명, 연락처는 표시하지 않는다.

### 갤러리

- `작업 전`, `작업 과정`, `작업 후`, `세부`로 분류한다.
- 각 섹션에는 공개 설정된 사진만 표시한다.
- 사진을 누르면 전체 화면 라이트박스를 연다.
- 라이트박스는 확대, 이전·다음, 현재 번호, 캡션, 닫기를 지원한다.
- 모바일 스와이프와 데스크톱 키보드 이동을 지원한다.
- 이미지가 1장인 그룹은 과도한 갤러리 UI를 만들지 않는다.

### 관련 사례

- 같은 서비스의 공개 사례를 우선한다.
- 현재 사례는 제외한다.
- 최대 3~4개
- 추천 알고리즘은 사용하지 않고 대표 우선·최신순 또는 공통 태그 기반의 단순 쿼리를 사용한다.

## 12.6 가격 안내 페이지

### 목적

서비스별 가격 정책을 한 곳에서 비교하고, 실제 사례별 가격으로 오해하지 않게 한다.

### 섹션 순서

1. 가격 안내 소개
2. 화장실 청소 가격
3. 에어컨 청소 가격
4. 아파트 유리창 청소 가격
5. 상가 유리창 청소 가격
6. 추가 비용이 달라지는 조건
7. 사진 확인이 필요한 이유
8. 가격 관련 FAQ
9. 향후 견적 진입 영역

### UI 원칙

- 데스크톱: 표 또는 서비스별 가격 카드
- 모바일: 가로 스크롤 표를 피하고 서비스별 세로 카드
- `○만원부터`, 포함 범위, 변동 조건을 구분해 읽기 쉽게 한다.
- 개별 사례 페이지에는 사례 당시 가격을 넣지 않는다.
- 가격이 미확정인 서비스에도 빈 숫자나 `미정`을 노출하지 않도록 콘텐츠 플레이스홀더를 구현 단계에서 교체한다.

# 13. 관리자 페이지 요구사항

## 13.1 관리자 로그인

- Supabase Auth 사용
- 고객은 별도 고객용 화면에서 회원가입할 수 있지만 관리자 회원가입 페이지는 제공하지 않는다.
- 관리자 계정은 기존 고객 계정에 슈퍼 관리자가 관리자 권한을 부여하는 방식으로 만든다.
- 초기 슈퍼 관리자 계정은 현재 코드·seed·브라우저 저장소에 만들지 않고 Supabase 연동 후 배포 운영 절차로 생성한다.
- 관리자 로그인은 이메일·비밀번호를 사용하며 계정 존재 여부나 역할을 구체적으로 노출하지 않는다.
- 인증되지 않은 사용자가 `/admin`에 접근하면 로그인으로 이동한다.
- 인증 상태, 활성 여부와 관리자 권한 검증은 서버에서 수행한다.
- 일반 관리자와 고객은 관리자 권한 부여·회수 기능에 접근할 수 없다.

## 13.1.1 고객 인증

- 고객은 이메일과 비밀번호로 회원가입·로그인·로그아웃할 수 있다.
- 고객은 계정 존재 여부를 노출하지 않는 비밀번호 재설정 요청을 사용할 수 있다.
- 고객 인증 화면과 관리자 인증 화면은 경로와 안내를 분리한다.
- 소셜 로그인, 결제, 예약, 리뷰와 상세 마이페이지는 이번 범위에 포함하지 않는다.

## 13.1.2 슈퍼 관리자 권한 관리

- `super_admin`만 기존 계정에 `admin` 권한을 부여하거나 회수한다.
- 마지막 활성 슈퍼 관리자의 슈퍼 권한 제거는 차단한다.
- 역할 변경은 대상, 이전·새 역할, 수행자, 시각과 결과를 감사 기록에 남긴다.
- Supabase 연동 전에는 슈퍼 계정, 관리자 seed와 역할 변경 성공 동작을 만들지 않는다.
- 관리자 사용자 관리 화면은 연동 전 잠금 상태와 필요한 설정만 안내한다.

## 13.5 사진 업로드

### 수량 제한

- 한 번의 파일 선택: 최대 20장
- 한 사례의 전체 사진 수: 최대 69장
- 여러 번에 나누어 업로드 가능
- 69장에 도달하면 추가 버튼 비활성화
- 업로드 전에 선택 수량과 남은 수량 검증

### 표시 예

```text
현재 38 / 69장
이번에 최대 20장까지 추가할 수 있습니다.
남은 업로드 가능 수량: 31장
```

### 허용 형식

- JPEG
- PNG
- WebP
- HEIC/HEIF는 모바일 업로드에서 수용하되 공개 가능한 웹 이미지로 변환한다.
- RAW 및 Live Photo 묶음은 MVP에서 지원하지 않는다.

### 파일당 제한

- 구현 기본값: 최대 20MB
- 제한 값은 환경 설정 또는 공통 상수로 관리한다.
- 손상 파일과 허용되지 않은 형식은 업로드 전에 안내한다.

### 업로드 UX

- 개별 파일 진행률
- 전체 진행률
- 성공, 업로드 중, 실패 상태
- 실패 파일만 재시도
- 일부 실패가 전체 성공 파일을 취소하지 않음
- 업로드 중 화면 이탈 경고
- 완료된 업로드는 화면 재접속 후 유지
- 동일 파일명은 허용하되 스토리지 경로는 UUID로 충돌 방지
- 가능한 경우 중복 파일 선택 경고

### 네트워크 안정성

모바일 네트워크 중단을 고려해 재개 가능한 업로드 방식을 사용한다.

### 고객 표시

- 공개 링크만 표시
- 영상이 없으면 섹션 전체 숨김
- 재생 전에는 썸네일과 재생 버튼만 표시
- 클릭 시 반응형 플레이어 로딩
- 자동 재생하지 않음
- 기본 비율 16:9
- Shorts도 레이아웃이 깨지지 않아야 한다.

## 13.9 공개 전 검토

공개 버튼을 누르기 전에 다음을 모두 확인해야 한다.

- 얼굴이나 식별 가능한 신체가 노출되지 않는다.
- 차량번호가 노출되지 않는다.
- 주소, 동·호수, 연락처가 노출되지 않는다.
- 개인 문서와 민감한 고객 물품이 노출되지 않는다.
- 공개 가능한 사진만 `공개`로 선택했다.
- 제목, 서비스, 지역, 대표 이미지가 설정됐다.
- 공개 사진이 최소 1장 존재한다.

체크가 완료되지 않으면 공개할 수 없다.

## 13.12 홈페이지 콘텐츠 관리

관리자는 개발자 도움 없이 고객용 홈페이지의 콘텐츠와 노출 설정을 관리할 수 있다. 페이지 템플릿, 섹션 종류, 그리드, 색상, 글꼴과 컴포넌트 스타일은 관리자 편집 범위에 포함하지 않는다.

### 관리자 내비게이션

- 대시보드
- 홈페이지 관리
  - 홈
  - 서비스
  - 가격 안내
  - 작업 과정
  - 업체 소개
  - 개인정보처리방침
- 사례 관리
- 미디어 보관함
- 사이트 설정

### 공통 편집 흐름

1. 관리자가 콘텐츠를 수정한다.
2. 변경 내용은 고객에게 보이지 않는 초안으로 저장된다.
3. 필수값, 이미지 대체 텍스트, 링크와 노출 규칙을 검증한다.
4. 관리자가 모바일·데스크톱 미리보기를 확인한다.
5. 게시 시 관련 고객 화면에 완전한 한 버전으로 반영한다.
6. 게시 실패 시 기존 게시 버전을 유지하고 초안과 재시도 행동을 보존한다.

### 홈 관리

- 히어로 eyebrow, 제목, 설명
- Primary·Secondary CTA 라벨과 링크
- 대표 이미지 업로드·선택·교체·삭제
- 이미지 대체 텍스트, 캡션, 초점 위치
- 서비스, 대표 사례, 최근 사례, 작업 과정, 가격, 신뢰 요소 섹션의 제목·설명
- 고정된 섹션의 노출 여부와 허용된 범위의 순서
- 홈 대표 사례 선택과 해제

### 서비스 관리

- 서비스명, 요약, 본문, 대표 이미지
- 작업 범위, 가격 요약, FAQ
- 정렬 순서와 활성 상태
- 네 서비스의 내부 식별자와 기존 사례 연결 관계는 일반 콘텐츠 편집으로 변경할 수 없다.
- 서비스 비활성화 전 연결된 공개 사례와 고객 화면 영향을 안내한다.

### 가격 관리

- 서비스별 가격 항목 생성·수정·삭제·정렬
- 표시 가격 또는 상담형 안내 문구
- 단위·범위, 포함·제외 사항, 가격 변동 조건
- 가격 페이지 상단 안내, 공통 주의 문구, FAQ
- 개별 포트폴리오 사례에는 당시 작업 가격을 표시하지 않는다.

### 안내 페이지 관리

- 작업 과정 단계의 생성·수정·삭제·정렬·노출 설정
- 업체 소개 문구, 가치 항목, 작업팀·장비 이미지, CTA
- 개인정보처리방침 본문과 게시 전 법률 검토 확인

### 사이트 설정

- 로고와 상호
- 대표 연락처와 문의 링크
- 운영 지역과 사업자 정보
- SNS 및 허용된 외부 링크
- 기본 SEO 제목·설명과 공유 이미지
- 헤더·푸터에 노출되는 공통 문구

### 편집 제한

- 페이지 템플릿과 섹션 종류를 임의로 추가하지 않는다.
- 색상, 글꼴, 간격, 반응형 그리드와 애니메이션을 변경하지 않는다.
- 시스템 필수 경로와 사례 개인정보 공개 규칙을 변경하지 않는다.
- 고객 화면에 스크립트, 임의 HTML 또는 검증되지 않은 외부 리소스를 삽입하지 않는다.

## 13.13 사이트 미디어 보관함

- 홈페이지용 이미지와 사례 이미지를 용도별로 구분한다.
- 업로드 진행률, 성공·실패 상태와 실패 항목 재시도를 제공한다.
- 이미지별 대체 텍스트, 캡션, 초점 위치와 사용 위치를 표시한다.
- 이미 사용 중인 이미지는 영향받는 페이지를 안내하고 교체 또는 사용 해제 전 삭제를 차단한다.
- 사용되지 않는 이미지는 확인 후 삭제할 수 있다.
- 목업 단계의 선택 파일은 세션 미리보기만 유지하고, 실제 운영 단계에서는 인증된 관리자용 원격 저장으로 교체한다.

## 13.14 콘텐츠 버전과 복원

- 초안과 게시 버전을 분리한다.
- 콘텐츠별 최근 수정 시각, 게시 시각과 수정 주체를 표시한다.
- 최근 게시 이력을 확인하고 선택한 버전을 새 초안으로 복원할 수 있다.
- 다른 관리자 또는 세션이 먼저 수정한 경우 충돌을 알리고 비교 또는 새로고침 후 진행하게 한다.
- 삭제·숨김·비활성화처럼 고객 노출에 영향을 주는 동작은 영향 범위와 확인 절차를 제공한다.

## 14.3 SEO 기능

- **FR-S-001:** 서비스마다 고유 title, description, canonical을 제공한다.

- **FR-S-006:** 서비스와 사례 페이지는 서버가 생성한 HTML에 핵심 본문을 포함한다.

- **FR-S-008:** 적절한 LocalBusiness, Service, BreadcrumbList, ImageObject 및 필요 시 VideoObject 구조화 데이터를 제공한다.
- **FR-S-009:** 이미지 alt 텍스트를 제공한다.
- **FR-S-010:** 천안·아산 지역명은 실제 콘텐츠 맥락에서 자연스럽게 사용하며 키워드 반복을 금지한다.

# 18. 애플리케이션 아키텍처

## 18.1 기본 원칙

- 고객용 핵심 콘텐츠는 Server Component를 기본으로 한다.
- 상호작용이 필요한 부분만 Client Component로 분리한다.
- 관리자 인증 페이지는 동적 렌더링한다.
- 공개 서비스와 사례 상세페이지는 정적 생성 또는 ISR 방식으로 빠르게 제공한다.
- DB 스키마는 SQL migration으로 관리한다.
- 브라우저에서 service role 키를 사용하지 않는다.

## 18.2 Client Component가 필요한 영역

- 사례 필터
- 더 보기
- 빠른보기 모달
- 라이트박스
- 전후 비교 슬라이더
- 모바일 스와이프
- 플로팅 `+` 버튼
- 관리자 폼
- 업로드 진행률
- 사진 다중 선택과 정렬
- YouTube 지연 로딩

나머지 설명 콘텐츠와 사례 본문은 서버 렌더링한다.

## 18.3 라우팅 구조 개념

```text
app/
├─ (site)/
```

# 20. Supabase Storage 설계

## 20.1 버킷

### `portfolio-originals`

- Private
- 관리자 업로드 원본
- 공개 전 및 내부 보관 자료
- 관리자만 접근

### `portfolio-public`

- Public 또는 안정적인 공개 전달이 가능한 정책
- 공개 검토를 마친 이미지의 웹용 복사본
- 불변 UUID 경로 사용

MVP에는 고객 문의 기능이 없으므로 `inquiry-attachments` 버킷은 만들지 않아도 된다. 향후 견적 기능에서 별도 private 버킷으로 추가한다.

## 20.2 경로 예

```text
portfolio-originals/{case-id}/{media-id}/original.ext
portfolio-public/{case-id}/{media-id}/{version}.webp
```

같은 공개 파일 경로를 덮어쓰지 않고 새 버전 경로를 사용한다.

## 20.3 공개 처리

- 원본은 private 버킷에 유지한다.
- 공개 대상으로 선택한 사진만 웹용 형식과 크기로 준비한다.
- EXIF 등 불필요한 메타데이터를 공개본에서 제거한다.
- 공개본은 검색과 캐시가 가능한 안정적인 URL을 사용한다.
- 비공개 전환 시 공개본을 삭제하거나 접근 불가능하게 하고 DB 경로를 제거한다.

## 20.4 업로드 경로

브라우저에서 Supabase Storage로 직접 업로드한다. 대용량 파일을 Next.js 서버를 통해 중계하지 않는다.

Next.js 서버는 다음만 담당한다.

- 관리자 인증 확인
- 업로드 가능 사례 확인
- 남은 사진 수 검증
- 업로드 경로 또는 권한 발급
- 업로드 완료 기록 검증

# 21. 이미지 처리

## 21.1 고객 전달

- Next.js Image 사용
- 반응형 sizes 정의
- 대표 이미지를 제외한 사진 lazy loading
- 화면 밖 이미지는 필요할 때 로딩
- width와 height를 DB에 저장하여 레이아웃 이동 방지
- 공개용 이미지에 적절한 압축과 최대 크기 적용
- 원본을 고객 페이지에 그대로 전달하지 않는다.

## 21.2 HEIC/HEIF

휴대전화 관리자 사용을 고려하여 HEIC/HEIF 선택을 허용한다.

- 업로드 또는 공개 처리 단계에서 JPEG/WebP 등 브라우저 호환 형식으로 변환
- 변환 실패 시 원본을 무조건 공개하지 않고 관리자에게 오류 표시
- 변환 구현 라이브러리는 후속 기술 명세에서 확정

## 21.3 이미지 최적화 제공자

MVP 기본값:

- Supabase Storage에 공개본 저장
- Next.js Image 및 Vercel Image Optimization으로 사용자 기기별 전달

동일 요청에서 Supabase 이미지 변환과 Vercel 이미지 변환을 중복 사용하지 않는다.

# 22. SEO 요구사항

## 22.1 URL

- 영문 소문자 slug
- 단어 구분은 하이픈
- slug는 공개 후 가급적 변경하지 않는다.
- 변경 시 리다이렉트 전략을 사용한다.

예:

```text
/services/bathroom-cleaning
/portfolio/cheonan-commercial-bathroom-cleaning
```

## 22.2 Metadata

각 서비스 및 공개 사례에 다음을 제공한다.

- title
- meta description
- canonical
- Open Graph title
- Open Graph description
- Open Graph image
- robots

관리자 입력이 없으면 데이터 기반 규칙으로 자동 생성한다.

## 22.3 Sitemap

포함:

- 홈
- 네 서비스
- 전체 사례
- 공개 사례
- 가격
- 작업 과정
- 업체 소개

제외:

- 관리자
- 필터 조합 URL
- 비공개 사례
- 삭제 사례
- 404
- 모달 전용 별도 URL은 존재하지 않음

## 22.4 Robots

- 관리자 경로 크롤링 차단
- 비공개 데이터는 robots에 의존하지 않고 서버와 DB에서 차단
- 필터 조합은 noindex 또는 canonical을 기본 서비스 페이지로 지정

## 22.5 구조화 데이터

- 홈/업체 소개: LocalBusiness 또는 적절한 Organization
- 서비스 페이지: Service
- 서비스 및 사례: BreadcrumbList
- 사례 이미지: ImageObject
- YouTube 영상이 있는 사례: VideoObject

실제 존재하지 않는 후기 점수, 가격, 주소, 자격 정보를 생성하지 않는다.

## 22.6 지역 콘텐츠

- 천안·아산을 서비스 설명과 실제 사례 맥락에 자연스럽게 사용한다.
- 지역명 반복 삽입을 금지한다.
- 실제 사례가 없는 세부 지역 랜딩페이지를 생성하지 않는다.

## 22.7 이미지 SEO

- 파일명보다 DB alt 텍스트와 주변 문맥을 우선 관리한다.
- alt는 장면을 설명한다.
- 키워드 나열을 금지한다.

좋은 예:

`청소 전 물때가 남은 천안 아파트 베란다 유리창`

나쁜 예:

`천안청소 천안유리창청소 유리창업체`

# 23. 성능 요구사항

사진 중심 제품이므로 성능은 MVP 필수 조건이다.

## 23.1 목표

실제 운영 데이터와 모바일 환경에서 다음을 목표로 한다.

- LCP: 2.5초 이하 목표
- CLS: 0.1 이하 목표
- INP: 200ms 이하 목표
- 첫 화면에 필요하지 않은 사진과 YouTube 플레이어는 지연 로딩
- 서비스 페이지 최초 데이터는 제한된 수량만 제공

이는 목표치이며 테스트 환경과 네트워크 조건을 함께 기록한다.

## 23.2 성능 규칙

- 히어로 대표 이미지 1장만 높은 우선순위로 로딩
- 모든 썸네일에 sizes 설정
- YouTube iframe 초기 로딩 금지
- 불필요한 Client Component 금지
- 전체 갤러리 데이터를 첫 요청에 모두 전달하지 않음
- 긴 목록은 cursor 페이지네이션
- 모달의 핵심 이미지 일부만 선로딩
- 애니메이션 라이브러리 남용 금지
- 번들 크기 증가를 유발하는 라이브러리는 사용 이유 기록

# 24. 접근성 요구사항

- 키보드만으로 주요 내비게이션, 카드, 필터, 모달, 라이트박스를 사용할 수 있어야 한다.
- 시맨틱 HTML을 사용한다.

- 터치 대상은 충분한 크기를 가진다.
- 오류 메시지는 입력 필드와 연결한다.
- 업로드 진행률과 실패 상태를 텍스트로도 알린다.
- 동작 감소 설정을 존중한다.

# 25. 보안 및 개인정보

## 25.1 RLS

공개 스키마의 모든 관련 테이블에 RLS를 활성화한다.

### 익명 사용자

- 공개 조건을 만족하는 사례, 사진, 영상, 서비스만 읽기 가능
- 생성·수정·삭제 불가

### 인증 관리자

- 활성 관리자 프로필이 확인된 사용자만 관리자 CRUD 가능
- Storage 정책도 동일한 관리자 검증 적용

## 25.2 키 관리

- Supabase publishable key는 허용된 공개 정책과 함께 사용
- service role 키는 server-only
- 브라우저 번들에 service role 및 비밀 환경변수 포함 금지
- 환경변수 유효성을 시작 시 검증

## 25.3 비공개 사례

- 단순히 UI에서 숨기는 것으로 끝내지 않는다.
- 공개 DB 쿼리와 RLS에서 차단한다.
- public Storage URL을 발급하지 않는다.
- 직접 URL 요청은 404
- 사이트맵 제외

## 25.4 관리자 입력

- 서버에서 Zod 검증
- slug, YouTube URL, 텍스트 길이 검증
- 파일 MIME 및 실제 파일 형식 검증
- 업로드 수량 검증을 클라이언트와 서버 양쪽에서 수행
- 관리자 본문은 허용된 텍스트 포맷만 사용
- 임의 HTML 입력을 기본적으로 허용하지 않는다.

## 25.5 개인정보 운영

- 공개 전 체크리스트 필수
- 정확한 고객 주소, 동·호수, 연락처 저장과 공개 필드 분리
- 고객용 `location_display`는 시·구·읍·면·동 수준의 안전한 표시값만 사용
- 내부 메모가 필요한 경우 공개 사례 테이블과 별도 비공개 구조 사용을 후속 명세에서 검토한다.

# 29. 화면별 인수 기준

## 29.1 홈

- 네 서비스 카드가 첫 주요 콘텐츠 영역에 보인다.
- 모바일에서 2×2로 깨지지 않는다.
- 서비스 카드 전체가 클릭 가능하다.
- 가격, 사례, 작업 과정 진입이 존재한다.
- 플로팅 `+` 버튼이 본문을 과도하게 가리지 않는다.

## 29.2 서비스 페이지

- 서비스 고유 H1과 설명이 있다.
- 공개 사례만 표시된다.
- 필터와 더 보기가 동작한다.
- 긴 목록 뒤에도 작업 과정, 가격, FAQ에 도달할 수 있다.
- 스티키 탭이 콘텐츠를 가리지 않는다.

## 29.3 빠른보기 모달

- URL이 사례 URL로 변경된다.
- 새로고침 시 상세페이지가 열린다.
- 뒤로가기로 모달이 닫힌다.
- 포커스와 스크롤 위치가 복원된다.
- 모바일에서 전체 화면으로 사용 가능하다.
- 모달 위에 플로팅 버튼이 겹치지 않는다.

## 29.4 사례 상세

- 공개 사진을 단계별로 보여준다.
- 빈 섹션을 숨긴다.
- 라이트박스가 키보드와 터치에서 동작한다.
- YouTube는 클릭 전 iframe을 로딩하지 않는다.
- 관련 사례가 현재 사례를 포함하지 않는다.
- 고유 메타데이터와 canonical이 있다.

## 29.5 관리자 업로드

- 모바일에서 카메라 롤 다중 선택이 가능하다.
- 20장 초과 선택을 업로드 전에 막는다.
- 전체 69장 제한을 서버에서도 검증한다.
- 업로드 일부 실패가 성공한 파일을 제거하지 않는다.
- 화면을 다시 열어도 완료 사진이 남아 있다.

## 29.6 공개 상태

- 신규 사례는 비공개다.
- 체크리스트 전에는 공개할 수 없다.
- 비공개 사례가 고객 목록, 개수, 사이트맵, 관련 사례에 없다.
- 비공개 직접 URL은 일반 404다.
- 비공개 사진 때문에 갤러리에 빈칸이나 번호 누락이 생기지 않는다.

## 29.7 홈페이지 콘텐츠 관리

- 관리자는 홈, 서비스, 가격, 작업 과정, 업체 소개, 개인정보처리방침과 공통 사이트 정보를 코드 수정 없이 편집한다.
- 초안 저장은 고객 화면을 바꾸지 않으며 게시 완료 후에만 새 버전이 노출된다.
- 홈 대표 이미지와 사이트 이미지를 업로드·교체·삭제할 수 있고 대체 텍스트 누락 시 게시할 수 없다.
- 가격 항목을 생성·수정·삭제·정렬할 수 있으며 개별 사례에는 당시 작업 가격이 표시되지 않는다.
- 사용 중인 이미지 삭제는 사용 위치 안내와 함께 차단된다.
- 관리자 미리보기는 390px 모바일과 1440px 데스크톱 상태를 제공한다.
- 게시 실패 시 기존 게시 버전과 편집 중 초안이 모두 보존된다.
- 이전 게시 버전을 새 초안으로 복원한 뒤 다시 검토하고 게시할 수 있다.
- 관리자에서는 콘텐츠와 노출 설정만 수정하며 페이지 레이아웃과 디자인 시스템을 변경할 수 없다.

# 32. 구현 원칙

하위 에이전트는 다음 규칙을 따라야 한다.

1. App Router만 사용한다.
2. Pages Router를 혼합하지 않는다.
3. 핵심 공개 콘텐츠를 클라이언트 useEffect로만 불러오지 않는다.
4. 서비스 페이지와 사례 상세페이지는 서버 렌더링 콘텐츠를 가진다.
5. 모달은 단순 로컬 상태만이 아니라 URL과 브라우저 히스토리를 반영한다.
6. DB 변경은 SQL migration으로 기록한다.
7. 생성된 Supabase Database 타입을 사용한다.
8. 새 테이블은 RLS와 정책을 함께 작성한다.
9. service role은 server-only다.
10. 사진은 브라우저에서 Storage로 직접 업로드한다.
11. 모든 사진에 너비와 높이를 기록한다.
12. 비공개 여부는 UI가 아니라 DB 정책과 서버 쿼리에서 강제한다.
13. 새 라이브러리는 기존 스택으로 해결하기 어려울 때만 추가한다.
14. 구현 완료 주장 전 lint, typecheck, test, e2e, build를 실행한다.
15. 기능 요구사항 ID를 커밋·작업 문서·테스트 이름과 연결할 수 있게 한다.

# 36. 최종 제품 흐름 요약

## 고객

```text
홈
→ 네 서비스 중 하나 선택
→ 해당 서비스의 여러 공개 사례 탐색
→ 카드 클릭
→ 빠른보기 모달
→ 전체 상세페이지
→ 전·과정·후 사진 및 YouTube 확인
→ 가격 안내 또는 향후 사진 견적 버튼
```

## 관리자

```text
모바일 로그인
→ 비공개 현장 생성
→ 한 번에 최대 20장 업로드
→ 여러 번 추가, 총 최대 69장
→ 사진 단계·순서·공개 여부 정리
→ 설명 및 YouTube 링크 입력
→ 개인정보 체크
→ 공개
→ 고객 페이지와 검색 대상에 반영
```

## 공개 규칙

```text
공개 사례 + 공개 사진만 고객에게 존재
비공개 사례와 사진은 목록·개수·URL·사이트맵에서 완전히 제외
```

# 37. 최종 인수 선언

다음 문장이 모두 참일 때 MVP를 제품 요구사항 관점에서 완료로 본다.

- 고객은 홈에서 네 서비스를 즉시 선택할 수 있다.
- 고객은 서비스별로 많은 사례를 사진 중심으로 탐색할 수 있다.
- 빠른보기 모달과 독립 상세페이지가 동일한 사례 URL을 자연스럽게 공유한다.
- 모바일과 데스크톱에서 주요 탐색 흐름이 모두 동작한다.
- 관리자는 휴대전화에서 안정적으로 여러 사진을 업로드하고 이어서 편집할 수 있다.
- 사진 수량 제한과 공개 전 개인정보 검토가 강제된다.
- 비공개 자료는 고객에게 존재하지 않는 것처럼 처리된다.
- YouTube 링크만으로 영상이 지연 로딩된다.
- 개별 사례 가격은 노출하지 않고 별도 가격 안내 흐름을 제공한다.
- 검색엔진이 공개 서비스와 사례의 핵심 콘텐츠를 읽을 수 있다.
- RLS, Storage 정책, 서버 검증으로 공개·비공개 경계를 보장한다.
- 핵심 테스트와 프로덕션 빌드가 통과한다.
